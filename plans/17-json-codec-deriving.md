# Plan 17: JSON Deriving â€” Haskell-Style Separate Encoder/Decoder Generation

## Priority: CRITICAL
## Effort: Medium (3-5 days) â€” infrastructure already exists in Port.hs
## Risk: Low â€” reuses battle-tested Port.hs machinery

## Problem

Canopy users have `elm/json` (`Json.Decode`, `Json.Encode`) but must write manual decoders and encoders for every type. This is the single biggest source of boilerplate in any Canopy application.

A 5-field record requires ~20 lines of mechanical code that mirrors the type definition:

```canopy
type alias User =
    { name : String
    , age : Int
    , email : String
    , role : String
    , active : Bool
    }

userDecoder : Json.Decode.Decoder User
userDecoder =
    Json.Decode.map5 User
        (Json.Decode.field "name" Json.Decode.string)
        (Json.Decode.field "age" Json.Decode.int)
        (Json.Decode.field "email" Json.Decode.string)
        (Json.Decode.field "role" Json.Decode.string)
        (Json.Decode.field "active" Json.Decode.bool)

encodeUser : User -> Json.Encode.Value
encodeUser user =
    Json.Encode.object
        [ ( "name", Json.Encode.string user.name )
        , ( "age", Json.Encode.int user.age )
        , ( "email", Json.Encode.string user.email )
        , ( "role", Json.Encode.string user.role )
        , ( "active", Json.Encode.bool user.active )
        ]
```

### Additional problems:
- `Json.Decode` only provides `map` through `map8` â€” records >8 fields require `json-decode-pipeline`
- Encoder and decoder are written separately â€” adding a field but forgetting one side is a runtime error
- Sum types require verbose `andThen` + tag dispatch â€” 15+ lines per type
- Nested types require manual decoder threading

### Key Discovery: The Infrastructure Already Exists

**`Optimize/Port.hs`** already generates JSON encoders and decoders from `Can.Type`:

```haskell
toEncoder :: Can.Type -> Names.Tracker Opt.Expr   -- Type â†’ encoder expression
toDecoder :: Can.Type -> Names.Tracker Opt.Expr   -- Type â†’ decoder expression
```

This supports: Int, Float, Bool, String, Maybe, List, Array, tuples, records, Json.Value.
It's used by the ports system and is battle-tested. **We just need to expose it to users.**

## Design: Haskell-Style `deriving`

Following Haskell's aeson pattern with separate `ToJSON`/`FromJSON`:

```canopy
type alias User =
    { name : String
    , age : Int
    , email : Maybe String
    , active : Bool
    }
    deriving (Json.Encode, Json.Decode)
```

The compiler generates two top-level functions:

```canopy
-- Generated by compiler:
encodeUser : User -> Json.Encode.Value
encodeUser user =
    Json.Encode.object
        [ ( "name", Json.Encode.string user.name )
        , ( "age", Json.Encode.int user.age )
        , ( "email", Maybe.withDefault Json.Encode.null (Maybe.map Json.Encode.string user.email) )
        , ( "active", Json.Encode.bool user.active )
        ]

userDecoder : Json.Decode.Decoder User
userDecoder =
    Json.Decode.map4 User
        (Json.Decode.field "name" Json.Decode.string)
        (Json.Decode.field "age" Json.Decode.int)
        (Json.Decode.field "email" (Json.Decode.nullable Json.Decode.string))
        (Json.Decode.field "active" Json.Decode.bool)
```

### Why Separate, Not Codec

Following Haskell's design (separate `ToJSON`/`FromJSON` rather than unified `Codec`):

1. **Matches existing API** â€” users already know `Json.Encode.Value` and `Json.Decode.Decoder`. Generated functions return these exact types. No new concepts.

2. **One-directional use cases** â€” you often only need encoding (sending to API) or only decoding (receiving from API), not both. Separate deriving lets you choose.

3. **Composability** â€” generated decoders compose with all existing `Json.Decode` combinators (`andThen`, `oneOf`, `maybe`). A custom `Codec` type would need its own combinator library.

4. **Incremental adoption** â€” you can derive the decoder but hand-write a custom encoder (or vice versa) when the JSON format doesn't match 1:1.

5. **Port.hs compatibility** â€” the existing infrastructure generates separate encoder/decoder expressions. A unified Codec would require new infrastructure.

### Naming Convention

| Type | Generated Encoder | Generated Decoder |
|------|-------------------|-------------------|
| `type alias User = ...` | `encodeUser : User -> Json.Encode.Value` | `userDecoder : Json.Decode.Decoder User` |
| `type alias ApiResponse = ...` | `encodeApiResponse : ApiResponse -> Json.Encode.Value` | `apiResponseDecoder : Json.Decode.Decoder ApiResponse` |
| `type Shape = ...` | `encodeShape : Shape -> Json.Encode.Value` | `shapeDecoder : Json.Decode.Decoder Shape` |

This follows Elm convention: decoders are `<type>Decoder`, encoders are `encode<Type>`.

### Deriving Only One Direction

```canopy
-- Decode only (e.g., reading API responses)
type alias ApiUser =
    { id : Int, name : String }
    deriving (Json.Decode)

-- Encode only (e.g., sending log events)
type alias AuditEvent =
    { action : String, timestamp : Int }
    deriving (Json.Encode)

-- Both directions
type alias User =
    { name : String, age : Int }
    deriving (Json.Encode, Json.Decode)
```

### Sum Types

```canopy
type Shape
    = Circle Float
    | Rectangle Float Float
    | Point
    deriving (Json.Encode, Json.Decode)
```

Generates tagged-object encoding (aeson `TaggedObject` default):

```json
{"tag": "Circle", "contents": 3.14}
{"tag": "Rectangle", "contents": [4.0, 5.0]}
{"tag": "Point"}
```

For all-nullary sum types (enums), generates string encoding (aeson `allNullaryToStringTag`):

```canopy
type Color = Red | Green | Blue
    deriving (Json.Encode, Json.Decode)
```

```json
"Red"
"Green"
"Blue"
```

### Options (via annotation)

For field renaming, optional fields, and sum type encoding strategy:

```canopy
type alias UserProfile =
    { userName : String
    , userEmail : String
    , userAvatar : Maybe String
    }
    deriving (Json.Encode { fieldNaming = SnakeCase, omitNothing = True }
             , Json.Decode { fieldNaming = SnakeCase, missingAsNothing = True })
```

JSON: `{"user_name": "Alice", "user_email": "a@b.com"}` (avatar omitted when Nothing)

**Supported options** (modeled after aeson's `Options`):

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `fieldNaming` | `CamelCase \| SnakeCase \| KebabCase \| Identity` | `Identity` | Field name transformation |
| `tagField` | `String` | `"tag"` | JSON key for constructor tag |
| `contentsField` | `String` | `"contents"` | JSON key for constructor payload |
| `omitNothing` | `Bool` | `False` | Omit `Maybe` fields that are `Nothing` from JSON |
| `missingAsNothing` | `Bool` | `False` | Treat missing JSON fields as `Nothing` |
| `unwrapSingle` | `Bool` | `False` | Unwrap single-field constructors |

### Parametric Types

```canopy
type alias Paginated a =
    { items : List a
    , total : Int
    , page : Int
    }
    deriving (Json.Encode, Json.Decode)
```

Generates functions that take encoder/decoder for the type parameter:

```canopy
-- Generated:
encodePaginated : (a -> Json.Encode.Value) -> Paginated a -> Json.Encode.Value

paginatedDecoder : Json.Decode.Decoder a -> Json.Decode.Decoder (Paginated a)
```

Usage:
```canopy
-- Concrete usage
Codec.decodeString (paginatedDecoder userDecoder) jsonString
```

This follows the exact pattern that `Json.Decode.list` uses â€” parametric decoders take a decoder argument.

### Nested Types

Just works. The compiler resolves inner types to their generated decoders:

```canopy
type alias Address =
    { street : String, city : String }
    deriving (Json.Encode, Json.Decode)

type alias Person =
    { name : String, address : Address }
    deriving (Json.Encode, Json.Decode)
```

`personDecoder` calls `addressDecoder` for the `address` field. If `Address` doesn't have a generated decoder, the compiler reports an error.

### Records >8 Fields

Port.hs currently caps at `map8`. For larger records, the compiler generates pipeline-style decoding using `Json.Decode.succeed` + `andThen`:

```canopy
-- For a 10-field record, generates:
bigRecordDecoder =
    Json.Decode.succeed BigRecord
        |> andMap (Json.Decode.field "field1" Json.Decode.string)
        |> andMap (Json.Decode.field "field2" Json.Decode.int)
        -- ... 8 more fields
```

Where `andMap` is:
```canopy
andMap : Json.Decode.Decoder a -> Json.Decode.Decoder (a -> b) -> Json.Decode.Decoder b
andMap = Json.Decode.map2 (|>)
```

This is the `elm-json-decode-pipeline` pattern, generated by the compiler.

## Implementation Plan

### Step 1: Add `deriving` Syntax to Parser

**File**: `packages/canopy-core/src/Parse/Declaration.hs`

After parsing a type alias or custom type, check for `deriving (...)`:

```haskell
data DerivingClause
  = DeriveJsonEncode !(Maybe JsonOptions)
  | DeriveJsonDecode !(Maybe JsonOptions)
  deriving (Eq, Show)

data JsonOptions = JsonOptions
  { _jsonFieldNaming :: !(Maybe NamingStrategy)
  , _jsonTagField :: !(Maybe Text)
  , _jsonContentsField :: !(Maybe Text)
  , _jsonOmitNothing :: !Bool
  , _jsonMissingAsNothing :: !Bool
  , _jsonUnwrapSingle :: !Bool
  }
```

**File**: `packages/canopy-core/src/AST/Source.hs`

Add `[DerivingClause]` to type alias and union definitions.

### Step 2: Carry Through Canonicalization

**File**: `packages/canopy-core/src/Canonicalize/Module.hs`

When canonicalizing types with `deriving`:
1. Record the deriving request in `Can.Alias` / `Can.Union`
2. Validate: all field types must be encodable/decodable (reject functions, type variables without constraints)

### Step 3: Extend Port.hs for Custom Types and Options

**File**: `packages/canopy-core/src/Optimize/Port.hs`

Currently supports: Int, Float, Bool, String, Maybe, List, Array, tuples, records, Json.Value.

Add:
- **Custom type (union) encoding/decoding** â€” tagged object format
- **All-nullary enum detection** â€” string encoding
- **JsonOptions support** â€” field naming, omitNothing, etc.
- **>8 field records** â€” pipeline-style decoding with `andMap`
- **Parametric type support** â€” generate functions that take encoder/decoder params

```haskell
toEncoderWithOptions :: JsonOptions -> Can.Type -> Names.Tracker Opt.Expr
toDecoderWithOptions :: JsonOptions -> Can.Type -> Names.Tracker Opt.Expr

toUnionEncoder :: JsonOptions -> Can.Union -> Names.Tracker Opt.Expr
toUnionDecoder :: JsonOptions -> Can.Union -> Names.Tracker Opt.Expr
```

### Step 4: Generate Definitions in Optimizer

**File**: `packages/canopy-core/src/Optimize/Module.hs`

After `addAliases` and `addUnions`, add:

```haskell
addJsonDeriving :: ModuleName.Canonical -> Can.Module -> Opt.LocalGraph -> Opt.LocalGraph
```

For each type with `deriving (Json.Encode)`:
1. Call `Port.toEncoderWithOptions` with the type
2. Create an `Opt.Define` node named `encode<TypeName>`
3. Add to the local graph

For each type with `deriving (Json.Decode)`:
1. Call `Port.toDecoderWithOptions` with the type
2. Create an `Opt.Define` node named `<typeName>Decoder`
3. Add to the local graph

### Step 5: Make Generated Functions Exportable

Generated functions need to:
- Appear in the module's value namespace
- Be exportable via `exposing (encodeUser, userDecoder)` or `exposing (..)`
- Be visible to the LSP for autocomplete
- Appear in generated docs

**File**: `packages/canopy-core/src/Canonicalize/Module.hs`

Add generated names to the module's `Can.Exports`.

### Step 6: `andMap` Helper for Large Records

**Option A**: Add `andMap` to `Json.Decode` in the `elm/json` package.

**Option B**: Generate it as an internal helper in the compiled output (no library change needed).

Prefer Option A â€” it's useful independently and matches `elm-json-decode-pipeline`.

### Step 7: Error Messages

When `deriving` fails (unsupported type, missing nested decoder):

```
-- DERIVING ERROR ---- src/Models.elm

Cannot derive `Json.Decode` for type `User`:

    14| type alias User =
    15|     { name : String
    16|     , callback : String -> Msg
                         ^^^^^^^^^^^^
    17|     }
    18|     deriving (Json.Decode)

The field `callback` has type `String -> Msg` which is a function.
JSON decoders cannot be generated for function types.

Hint: Remove `callback` from the record, or write a manual decoder
instead of using `deriving`.
```

### Step 8: Tests

- Record types: 1, 5, 8, 10, 20 fields
- Optional fields: `Maybe` fields with `omitNothing` and `missingAsNothing`
- Sum types: mixed constructors (nullary + record + positional)
- All-nullary sum types: string encoding
- Nested types: record containing record
- Parametric types: `Paginated a` generates `paginatedDecoder : Decoder a -> Decoder (Paginated a)`
- Field naming strategies: Identity, SnakeCase, CamelCase
- >8 field records: pipeline decoding
- One-directional deriving: only `Json.Encode` or only `Json.Decode`
- Error cases: function types, unconstrained type variables, extensible records
- Roundtrip: `decode (encode x) == Ok x` for all derivable types
- Compatibility: generated decoders compose with existing `Json.Decode` combinators
