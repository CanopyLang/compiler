# Plan 08: TEA at Scale

## Priority: MEDIUM — Tier 2
## Effort: 2-3 weeks (reduced from 4-5)
## Depends on: Plan 03 (packages), Plan 05 (CanopyKit for route code splitting)

> **Note (revised):** This plan was originally "State Architecture Beyond TEA" with a 4-5 week
> Stores API. It has been significantly simplified. TEA has been used successfully at Culture Amp
> (300K LOC), NoRedInk (500K LOC), and others. The "TEA fails at scale" claim was based on
> Redux/Zustand benchmarks in React, which measures a fundamentally different architecture.
>
> The actual pain points at scale are: (1) verbose nested record updates and (2) no route-level
> code splitting. Plan 26a solves the first. This plan solves the second.

## Problem

Two genuine TEA pain points at scale:

1. **No code splitting**: TEA has no story for lazy-loading routes. Large apps ship all route code upfront, leading to slow initial loads.
2. **Message indirection**: In large apps with deeply nested components, message delegation through multiple layers creates noise.

Both are solvable without introducing mutable state.

## Solution 1: Route-Level Code Splitting

CanopyKit (Plan 05) handles file-based routing. Each route module is compiled to a separate JS chunk and loaded on navigation:

```canopy
-- Generated by CanopyKit router:
-- Each page is a separate ESM module, loaded lazily
type Route
    = Home        -- loads ./Routes.Home.js on first visit
    | About       -- loads ./Routes.About.js on first visit
    | BlogPost { slug : String }  -- loads ./Routes.Blog.Post.js
```

The compiler emits dynamic `import()` calls for each route, giving browsers the ability to load only the code needed for the current page.

### Implementation

This is handled by Plan 05 (CanopyKit) and Plan 01 (ESM output). This plan documents the architecture but the implementation lives in those plans.

## Solution 2: Message Delegation Helpers

Provide standard library helpers for the common "parent delegates to child" pattern:

```canopy
module Platform.Delegate exposing (delegate)

{-| Transform a child's (model, Cmd msg) into a parent's (model, Cmd parentMsg).

    Eliminates the boilerplate of wrapping child messages at every level.
-}
delegate :
    { toModel : childModel -> parentModel -> parentModel
    , toMsg : childMsg -> parentMsg
    }
    -> ( childModel, Cmd childMsg )
    -> parentModel
    -> ( parentModel, Cmd parentMsg )
delegate config ( childModel, childCmd ) parentModel =
    ( config.toModel childModel parentModel
    , Cmd.map config.toMsg childCmd
    )
```

Usage:

```canopy
update msg model =
    case msg of
        SettingsMsg subMsg ->
            Settings.update subMsg model.settings
                |> delegate
                    { toModel = \s m -> { m | settings = s }
                    , toMsg = SettingsMsg
                    }
                    model
```

This is a library-level solution, not a compiler change. Include it in canopy/core or a new canopy/platform-helpers package.

## What This Plan Does NOT Include

**Stores/Signals are deferred.** The original plan proposed a `Store` type with mutable state behind a Cmd facade. This introduces hidden mutability — a philosophical departure from Elm that could alienate core users while not being different enough to attract React developers.

If real user demand emerges for cross-component reactive state (after CanopyKit ships and people build large apps), Stores can be revisited. But the evidence from Elm's production use suggests that nested record updates (Plan 26a) + route code splitting solve 80% of the pain.
