# Plan 26: Modern Web API Integration

## Priority: LOW -- Tier 4
## Status: ~80% complete
## Effort: 2-3 weeks (most web API bindings already exist as packages)
## Depends on: Plan 03 (packages -- COMPLETE), Plan 13 (capabilities -- core COMPLETE)

## What Already Exists

The Canopy ecosystem has **72 stdlib packages** covering most Web APIs. The following are already implemented:

### Browser and DOM
- `canopy/browser` (5 files) -- navigation, visibility, online/offline, viewport
- `canopy/html` -- DOM elements, attributes, events
- `canopy/svg` -- SVG elements and attributes
- `canopy/router` (4 files) -- client-side routing

### Storage and Data
- `canopy/indexed-db` (7 files) -- IndexedDB with schema versioning
- `canopy/json` -- JSON encoding/decoding
- `canopy/bytes` -- binary data handling

### Networking
- `canopy/http` -- fetch-based HTTP client
- `canopy/websocket` (4 files) -- WebSocket with reconnection
- `canopy/webrtc` -- WebRTC peer connections
- `canopy/web-worker` (4 files) -- Web Worker management and pooling

### Media and Hardware
- `canopy/camera` -- camera access
- `canopy/web-audio` -- Web Audio API
- `canopy/web-crypto` -- cryptographic operations
- `canopy/webauthn` -- WebAuthn/passkey authentication

### UI and Components
- `canopy/headless-ui` (10 files) -- accessible headless components
- `canopy/pwa` (7 files) -- Progressive Web App (service worker, manifest, offline)
- `canopy/web-component-export` (6 files) -- export Canopy components as Web Components
- `canopy/virtual-list` -- virtualized rendering for large lists

## What Remains

These are newer Web APIs that shipped in 2025-2026 and do not yet have Canopy bindings:

### Phase 1: Temporal API (Week 1)
- Canopy types for `PlainDate`, `ZonedDateTime`, `Duration`, `Instant`
- FFI bindings to the Temporal API (Chrome 144+, Firefox 139+)
- Type-safe formatting via `Intl.DateTimeFormat`
- Timezone-aware arithmetic
- Replaces the need for moment.js/date-fns entirely
- Capability: requires `temporal` (graceful fallback for older browsers)

### Phase 2: Navigation API + Popover API + CSS Anchor Positioning (Week 2)
- **Navigation API** (Baseline Jan 2026): CanopyKit router compiles to `navigation.navigate()`/`navigation.addEventListener("navigate", ...)`. Browser handles scroll restoration, focus management, accessibility announcements automatically.
- **Popover API** (Baseline Jan 2025): `popover` and `popoverTarget` attributes in Html module. Native light dismiss, Escape key, focus management, top layer (no z-index conflicts). Zero JS runtime needed.
- **CSS Anchor Positioning** (all major browsers): `anchorName`, `positionAnchor`, anchor functions in CSS module. Replaces Floating UI/Popper.js.

### Phase 3: Speculation Rules + Container Queries (Week 3)
- **Speculation Rules** (Chrome, expanding): CanopyKit generates `<script type="speculationrules">` from route structure. Automatic prefetch/prerender based on route analysis.
  ```html
  <!-- Auto-generated by CanopyKit build -->
  <script type="speculationrules">
  {
    "prerender": [
      { "where": { "href_matches": "/blog/*" }, "eagerness": "moderate" }
    ]
  }
  </script>
  ```
- **Container Queries** (widely available): `containerType`, `containerName` in CSS module. `@container` query syntax for component-scoped responsiveness.

## Definition of Done

- [ ] Temporal API types and FFI bindings with timezone support
- [ ] Navigation API integration in CanopyKit router
- [ ] Popover API attributes in Html module
- [ ] CSS Anchor Positioning in CSS module
- [ ] Speculation Rules generation from CanopyKit route structure
- [ ] Container Queries support in CSS module
- [ ] All new APIs gated behind capabilities for graceful degradation

## Risks

- **Browser support gaps**: Temporal API and Speculation Rules are not yet universal. Must provide capability-gated fallbacks.
- **API churn**: Some newer APIs may see spec changes before stabilization. Pin to specific browser versions in testing.
- **Polyfill strategy**: Decide whether to ship polyfills or just gate behind capability checks.
