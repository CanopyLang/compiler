## Problem

Vite has 98% developer retention (State of JS 2025). Developers expect:
- Dev server starts in < 200ms
- Code changes reflected in < 100ms (HMR)
- Source maps that work in browser DevTools
- Integration with the rest of their JS/TS project

Canopy currently has `canopy reactor` which is a custom dev server. This is fine for standalone Canopy apps, but the industry has converged on Vite.

## Solution: vite-plugin-canopy

A Vite plugin that makes `.can` files first-class in any Vite project.

### Usage

```javascript
// vite.config.js
import canopy from 'vite-plugin-canopy'

export default {
  plugins: [canopy()]
}
```

```javascript
// main.js — use Canopy from JavaScript/TypeScript
import { Elm } from './src/Main.can'

const app = Elm.Main.init({
  node: document.getElementById('app')
})
```

### How It Works

#### File Resolution

The plugin registers `.can` as a resolvable file type:

```javascript
{
  name: 'vite-plugin-canopy',

  resolveId(source) {
    if (source.endsWith('.can')) return source
  },

  async load(id) {
    if (!id.endsWith('.can')) return null
    // Invoke canopy compiler, return ESM output
    const result = await compileCanopy(id)
    return { code: result.js, map: result.sourceMap }
  }
}
```

#### Compilation

The plugin invokes the Canopy compiler as a long-running process (or uses the compiler as a library if possible). The compiler outputs ESM (from Plan 01), which Vite processes natively.

#### HMR Implementation

When a `.can` file changes:

1. **Vite detects the change** via chokidar
2. **Plugin recompiles** the changed module
3. **State capture**: Before swap, serialize the current model
4. **Module swap**: Vite replaces the module via `import.meta.hot`
5. **State restore**: Initialize the new module with the serialized model
6. **Model compatibility check**: If the model type changed incompatibly (detected by comparing model structure), fall back to full page reload

The HMR injection code:

```javascript
// Injected by the plugin into each compiled module
if (import.meta.hot) {
  import.meta.hot.accept((newModule) => {
    const oldModel = __canopy_getModel__();
    try {
      __canopy_hotSwap__(newModule, oldModel);
    } catch (e) {
      import.meta.hot.invalidate();  // full reload
    }
  });
}
```

#### Model Type Hashing

The compiler generates a hash of the Model type structure. When the model type changes, the hash changes, and the HMR code detects the mismatch:

```javascript
// Generated by compiler
export const __MODEL_HASH__ = "a7f3b2c1";  // hash of Model type structure

// HMR check
if (oldModule.__MODEL_HASH__ !== newModule.__MODEL_HASH__) {
  import.meta.hot.invalidate();  // model shape changed, full reload
}
```

### Error Overlay

When compilation fails, display Canopy's excellent error messages in Vite's error overlay:

```javascript
handleHotUpdate({ file, server }) {
  try {
    compile(file)
  } catch (error) {
    server.ws.send({
      type: 'error',
      err: {
        message: error.title,
        stack: error.details,  // Canopy's formatted error output
        loc: { file, line: error.line, column: error.column }
      }
    })
  }
}
```

## Implementation

### Package: `vite-plugin-canopy`

Written in TypeScript. Published to npm.

```
vite-plugin-canopy/
  src/
    index.ts          — Plugin entry, Vite plugin interface
    compiler.ts       — Canopy compiler process management
    hmr.ts            — HMR injection and state management
    errors.ts         — Error formatting for Vite overlay
    types.ts          — TypeScript types
  hmr-runtime.js      — Client-side HMR code injected into output
  package.json
  tsconfig.json
```

### Compiler Communication

Two modes:

**Mode A (Process):** Shell out to `canopy make` per compilation. Simple but slow. Good for initial implementation.

**Mode B (Server):** Run the Canopy compiler as a long-running process with a JSON-RPC or stdin/stdout protocol. Compiler keeps module graph in memory, recompiles incrementally. Required for < 100ms HMR.

Start with Mode A, move to Mode B when Plan 09 (incremental compilation) is ready.

### Development Workflow

```bash

cd my-app
npm run dev    # Starts Vite with canopy plugin
```

The `create-canopy-app` scaffolder generates:
- `vite.config.js` with the plugin
- `src/Main.can` starter app
- `index.html` with mount point
- `package.json` with dependencies

## Testing

- Unit tests for the plugin (Vitest)
- Integration test: create a Canopy app, start Vite, verify it renders
- HMR test: modify a .can file, verify the browser updates without full reload
- Error test: introduce a type error, verify the error overlay shows Canopy's message
- Compatibility: test with Vite 7+ and Vite 8 (Rolldown-powered)

## Dependencies

- Plan 01 (ESM output) — plugin needs ESM output to work with Vite
- `canopy` compiler binary must be available on PATH or configurable

## Risks

- Compiler startup time may make first compilation slow. Mitigate with Mode B (long-running process).
- Source maps: the compiler must produce valid v3 source maps that Vite can consume and transform.
