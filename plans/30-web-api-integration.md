# Plan 30: Modern Web API Integration

## Priority: MEDIUM — Tier 2
## Effort: 3-4 weeks
## Depends on: Plan 03 (packages — COMPLETE), Plan 13 (capabilities — core COMPLETE)

## Problem

The web platform has shipped transformative APIs in 2024-2026 that our plans don't leverage:

| API | Status | What It Replaces |
|-----|--------|-----------------|
| Navigation API | Baseline Jan 2026 | Custom router history management |
| View Transitions API | Baseline Oct 2025 | JS animation libraries for page transitions |
| Popover API | Baseline Jan 2025 | Modal/tooltip JS libraries |
| Container Queries | Widely Available | Media queries for components |
| Temporal API | Shipping (Chrome 144, Firefox 139) | Date, moment.js, date-fns |
| CSS Anchor Positioning | All major browsers | Floating UI, Popper.js |
| Speculation Rules | Chrome (expanding) | Manual prefetch/preload |

## Solution: First-Class Integration with New Web APIs

### Navigation API (Replaces Custom Router)

CanopyKit's router should compile to Navigation API calls:

```canopy
-- CanopyKit generates:
-- navigation.addEventListener("navigate", (event) => {
--   event.intercept({ handler: () => loadRoute(event.destination.url) });
-- });

-- Developer benefit: browser handles scroll restoration, focus management,
-- accessibility announcements automatically. Zero custom code needed.
```

### Temporal API (Replaces Date)

```canopy
module Canopy.Time.Temporal exposing (PlainDate, ZonedDateTime, Duration, Instant)

-- Type-safe date/time that compiles to Temporal API:
today : Task Never PlainDate
today = Temporal.now.plainDate "iso8601"

-- Duration arithmetic:
nextWeek : PlainDate -> PlainDate
nextWeek date = Temporal.add date (Duration.days 7)

-- Timezone-aware:
meetingTime : ZonedDateTime
meetingTime =
    Temporal.zonedDateTime
        { year = 2026, month = 3, day = 15
        , hour = 14, minute = 0
        , timeZone = "America/New_York"
        }

-- Format (type-safe, replaces Intl.DateTimeFormat):
formatted = Temporal.format meetingTime
    { dateStyle = Long, timeStyle = Short, locale = "en-US" }
-- "March 15, 2026 at 2:00 PM"
```

### Popover API (Native Popovers)

```canopy
-- Canopy generates native popover attributes:
button [ popoverTarget "menu" ] [ text "Open menu" ]
div [ popover Auto, id "menu" ]
    [ menuContent ]

-- No JS runtime needed. Browser handles:
-- - Light dismiss (click outside)
-- - Escape key
-- - Focus management
-- - Top layer (no z-index wars)
```

### CSS Anchor Positioning

```canopy
-- Type-safe anchor positioning in CSS:
tooltip model =
    div
        [ id "tooltip-anchor"
        , css [ anchorName "--tooltip-anchor" ]
        ]
        [ text model.label ]
    ++
    div
        [ css
            [ positionAnchor "--tooltip-anchor"
            , position Absolute
            , top (Anchor "--tooltip-anchor" Bottom)
            , left (Anchor "--tooltip-anchor" Center)
            ]
        , popover Auto
        ]
        [ text model.tooltipText ]
```

### Speculation Rules (Auto-Prefetch)

CanopyKit generates speculation rules from the route structure:

```html
<!-- Auto-generated by CanopyKit build -->
<script type="speculationrules">
{
  "prerender": [
    { "where": { "href_matches": "/blog/*" }, "eagerness": "moderate" }
  ],
  "prefetch": [
    { "where": { "selector_matches": "a[href]" }, "eagerness": "conservative" }
  ]
}
</script>
```

The compiler knows all routes and can generate optimal prefetch rules automatically.

## Implementation

### Phase 1: Temporal API types (Week 1)
- Canopy types for PlainDate, ZonedDateTime, Duration, Instant
- FFI bindings to Temporal API
- Formatting via Intl.DateTimeFormat
- Capability: requires `temporal` (graceful fallback for old browsers)

### Phase 2: Navigation + Popover + Anchor (Week 2)
- Navigation API integration in CanopyKit router
- Popover attributes in Html module
- Anchor positioning in CSS module
- All compile to native browser features — zero JS runtime

### Phase 3: Speculation Rules (Week 3)
- Route analysis during build
- Speculation rules JSON generation
- Eagerness heuristics based on component priority

### Phase 4: Container Queries (Week 4)
- `containerType` and `containerName` in CSS module
- `@container` query syntax in responsive API
- Component-scoped responsiveness
